                    1. COST MONITORING OVERVIEW
================================================================================

WHY COST MONITORING IS CRITICAL FOR HFT:
-----------------------------------------

In High-Frequency Trading, costs can spiral out of control quickly:
    - A forgotten cloud instance: $500/day
    - An inefficient trading algorithm: $5,000/day in excess fees
    - A data feed left running: $1,000/day
    - Poor order routing: $10,000/day in lost rebates

Without real-time monitoring, these issues can persist for days or weeks,
resulting in hundreds of thousands in unnecessary costs.

COST MONITORING PHILOSOPHY:
---------------------------

"What gets measured gets managed."

Our cost monitoring approach:
    1. REAL-TIME: Costs tracked every 60 seconds
    2. GRANULAR: Costs attributed to individual strategies
    3. PREDICTIVE: Anomalies detected before they impact P&L
    4. ACTIONABLE: Automated alerts with clear remediation steps
    5. TRANSPARENT: All teams can see their cost impact

COST MONITORING SCOPE:
----------------------

We monitor costs across 5 categories:

+------------------+-----------+--------------+--------------+

+------------------+-----------+--------------+--------------+

| Total            | $62,035   | 60 seconds   | +10%         |
+------------------+-----------+--------------+--------------+

Note: Trading costs refresh every 60 seconds because they're the most

DATA FLOW:
----------

1. Cost data sources emit events (every trade, every 5 minutes, etc.)
2. Events ingested into Kafka stream processing
3. Real-time aggregation and calculation
4. Parallel processing for:
   - Dashboard updates
   - Anomaly detection
   - Cost attribution
5. Alerts triggered if thresholds exceeded
6. Data stored in time-series database (InfluxDB) for analytics

Latency:
    Trading Cost Updates: <2 seconds
    Infrastructure Updates: <30 seconds
    Dashboard Refresh: Every 60 seconds
    Alert Delivery: <10 seconds from threshold breach

COST MONITORING METRICS:
-------------------------

Primary KPIs:
    1. Total Daily Cost vs Budget
    2. Cost per Trade
    3. Cost as % of Revenue
    4. Cost per Strategy
    5. Variance from Forecast

Secondary KPIs:
    6. Exchange Fee Efficiency (maker ratio)
    7. Data Feed Utilization
    8. Cloud Cost Efficiency (RI coverage, spot usage)
    9. Cost Growth Rate (WoW, MoM)
    10. Cost Anomaly Frequency

Target Metrics:
    - Daily Cost Variance: <5% from budget
    - Cost Anomalies: <2 per month
    - Alert Response Time: <15 minutes
    - Cost Attribution Accuracy: >95%
    - Dashboard Uptime: 99.9%

ALERT THRESHOLD FRAMEWORK:
--------------------------

We use a tiered alert system with escalating severity:

INFO (Informational):
    - 10-15% over budget
    - Email notification to team
    - Review within 24 hours

WARNING (Action Needed):
    - 15-25% over budget
    - Slack alert to team + manager
    - Review within 4 hours
    - Initiate investigation

CRITICAL (Immediate Action):
    - >25% over budget
    - Page on-call engineer (SMS)
    - Escalate to VP Engineering
    - Immediate investigation required
    - Consider emergency shutdown of non-critical systems

================================================================================
                    4. COST ATTRIBUTION METHODOLOGY
================================================================================

WHY COST ATTRIBUTION MATTERS:
------------------------------

Without accurate cost attribution:
    - Can't determine true profitability of individual strategies
    - No accountability for cost management
    - Difficult to identify optimization opportunities
    - Resource allocation decisions are uninformed

With accurate cost attribution:
    - Every strategy has a complete P&L including costs
    - Teams are incentivized to manage their costs
    - Easy to identify unprofitable strategies
    - Data-driven resource allocation

COST ATTRIBUTION HIERARCHY:
---------------------------

Level 1: Direct Costs (Easy to attribute)
    - Exchange fees for specific trades
    - Cloud resources tagged to specific projects
    - Dedicated market data feeds for specific strategies

Level 2: Allocated Costs (Proportional attribution)
    - Shared compute resources (by CPU/memory usage)
    - Shared network bandwidth (by data transfer volume)
    - Shared market data feeds (by symbols used)

Level 3: Overhead Costs (Evenly distributed)
    - Core infrastructure (monitoring, security)
    - Shared services (DNS, authentication)
    - Management overhead (executive costs)

ATTRIBUTION METHODOLOGY BY COST CATEGORY:
------------------------------------------

1. EXCHANGE FEES (Direct Attribution):
   Method: Tag each trade with strategy_id
   Attribution: 100% direct to strategy that generated trade
   Accuracy: 100%

   Example:
   ```
   Trade: Buy 100 SPY @ $450.00
   Strategy: MM_SPY_001
   Exchange Fee: $0.0028 per share Ã— 100 = $0.28
   Attributed To: MM_SPY_001
   ```

2. MARKET IMPACT & SLIPPAGE (Modeled Attribution):
   Method: Calculate market impact per trade using impact model
   Attribution: Direct to strategy, but estimated not measured
   Accuracy: 85-90% (model-based)

   Example:
   ```
   Trade: Buy 10,000 ABC @ $50.00
   Expected Price: $50.00
   Actual Average Fill: $50.0125
   Slippage: $0.0125 per share Ã— 10,000 = $125
   Attributed To: STAT_ABC_002
   ```

3. INFRASTRUCTURE COSTS (Proportional Attribution):
   Method: Allocate based on compute resource usage

   Metrics Used:
   - CPU time (core-seconds)
   - Memory usage (GB-seconds)
   - Network bandwidth (bytes transferred)
   - Storage usage (GB-hours)

   Calculation:
   ```
   Strategy_Share = (Strategy_CPU / Total_CPU) Ã— 0.40 +
                    (Strategy_Memory / Total_Memory) Ã— 0.30 +
                    (Strategy_Network / Total_Network) Ã— 0.20 +
                    (Strategy_Storage / Total_Storage) Ã— 0.10

   Strategy_Infra_Cost = Total_Infra_Cost Ã— Strategy_Share
   ```

   Accuracy: 75-80% (usage-based estimation)

4. MARKET DATA COSTS (Hybrid Attribution):
   Method: Direct for unique feeds, proportional for shared feeds

   Direct Attribution:
   - Strategy-specific data feeds (100% to that strategy)

   Proportional Attribution:
   - Shared feeds allocated by symbols used and message volume

   Example:
   ```
   NYSE Level 2 Feed: $14,000/month (shared by all strategies)

   Strategy MM_SPY_001:
   - Uses: SPY, QQQ, IWM, DIA (4 symbols)
   - Message Volume: 45,000 msg/sec
   - Total Symbols Tracked: 120
   - Total Message Volume: 180,000 msg/sec

   Attribution:
   Symbol Share: 4/120 = 3.33%
   Message Share: 45,000/180,000 = 25.0%

   Blended Share: (3.33% Ã— 0.40) + (25.0% Ã— 0.60) = 16.33%
   Attributed Cost: $14,000 Ã— 16.33% = $2,286/month = $75/day
   ```

   Accuracy: 70-75%

5. CLOUD COMPUTE COSTS (Tagged Attribution):
   Method: Tag all cloud resources with strategy/project

   AWS Resource Tagging:
   ```
   Tags:
   - Strategy: MM_SPY_001
   - Team: trading-desk
   - Environment: production
   - Cost-Center: trading
   ```

   Attribution: Direct to tagged strategy
   Accuracy: 95% (tags must be complete)

   Untagged Resources: Distributed proportionally

COST ATTRIBUTION ACCURACY VALIDATION:
-------------------------------------

We validate attribution accuracy by ensuring:

1. Sum of Attributed Costs = Total Costs
   ```
   Sum(All_Strategy_Costs) + Unallocated = Total_Daily_Cost
   ```

   Actual Check:
   Attributed to Strategies: $61,250
   Unallocated (shared): $785
   Total: $62,035 âœ“ (matches actual total)

2. No Negative Costs (except rebates)
   - All cost categories must be >= 0 except exchange rebates

3. Attribution Percentage Check
   - No strategy should have >50% of any shared cost
   - If so, indicates attribution error

Monthly Attribution Audit:
    - Compare attributed costs to invoices
    - Reconcile discrepancies
    - Adjust attribution models if needed
    - Target: <5% variance from actual
