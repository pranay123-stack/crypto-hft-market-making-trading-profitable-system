================================================================================
                         ALERT SEVERITY LEVELS
================================================================================

P1 - CRITICAL (Red)
- Response Time: 5 minutes
- Channels: PagerDuty + SMS + Slack + Email
- Escalation: Every 10 minutes
- Examples: System down, trading halted, security breach

P2 - HIGH (Orange)
- Response Time: 15 minutes
- Channels: PagerDuty + Slack + Email
- Escalation: After 30 minutes
- Examples: Degraded performance, high error rate, risk breach

P3 - MEDIUM (Yellow)
- Response Time: 1 hour
- Channels: Slack + Email
- Escalation: Manual only
- Examples: Minor issues, warnings, config drift

P4 - LOW (Green)
- Response Time: Best effort
- Channels: Email only
- Escalation: None
- Examples: Informational, reports, summaries

================================================================================
                         TROUBLESHOOTING GUIDE
================================================================================

Notifications Not Delivered:
1. Check service status: systemctl status hft-notification
2. Check channel health: curl http://localhost:8080/health
3. Check logs: tail -f /var/log/hft/notification.log
4. Verify credentials: ./scripts/test_notifications.sh
5. Check network: ping api.pagerduty.com

High Latency:
1. Check queue depth: redis-cli LLEN hft:notification:queue
2. Monitor CPU/Memory: top -p $(pidof notification_manager)
3. Check network latency: ping smtp.company.com
4. Review rate limiter: curl http://localhost:8080/metrics
5. Scale up instances if needed

Duplicate Notifications:
1. Check deduplication: redis-cli KEYS "dedup:*"
2. Verify single instance per alert
3. Review application code
4. Enable idempotency keys

Missing P1 Alerts:
1. Verify P1 bypass enabled
2. Check PagerDuty integration
3. Review routing rules
4. Check alert classification
5. Review recent P1 logs
